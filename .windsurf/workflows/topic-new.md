# TOPIC NEW — Create a New Documentation Topic

## 🌐 Project Override Hook

Before executing this workflow, check whether the host project provides an override.

**Path** (in the host project's workspace root, NOT this global repo):
`.windsurf/overrides/workflows/<same-filename-as-this-file>.md`

If present, apply its mode (frontmatter `mode:` OR first heading `# REPLACE` / `# EXTEND` / `# MODIFY`):

- **REPLACE** — run ONLY the override; ignore the steps below.
- **EXTEND** — run the steps below as written, then run the override as a final phase (or at the position it specifies).
- **MODIFY** — run the steps below, with the steps the override declares substituted / inserted at the matching positions.

No override → proceed as written. Ambiguous mode → **STOP and ask the user**.

---

> This is a **strict step-by-step workflow**, NOT background context.
> - Execute steps **in order**, one at a time
> - **Do NOT skip steps** — every step exists for a reason
> - **Do NOT combine or optimize** steps unless explicitly allowed

Creates a new topic folder under `docs/` with a standardized structure — the same structure used by the Angular migration topic and any future topics.

## Inputs

- **TOPIC_SLUG**: Folder name, e.g. `performance-optimization`, `accessibility-audit`
- **TOPIC_TITLE**: Human-readable title, e.g. "Performance Optimization"
- **TOPIC_DESCRIPTION**: One-line description of what this topic covers

## Topic Structure

```
docs/{TOPIC_SLUG}/
├── README.md             ← topic overview, status table, links
├── CHANGELOG.md          ← per-topic date-grouped changelog (append on each session)
├── specs/                ← canonical 5-spec set (see spec-author skill)
├── tickets/              ← ticket specs (one per unit of work)
├── reports/              ← per-ticket completion + verification reports
├── research/             ← raw research artifacts (optional — only if research
│                        happened in Cascade; Traycer-produced research may
│                        live elsewhere and be referenced)
└── stakeholder-reports/  ← non-technical stakeholder updates (optional —
                            created on demand by /stakeholder-report; never
                            auto-generated by the topic lifecycle)
```

**Convention references:**
- `docs/HOW_TO_USE.md` — authoritative rules for the doc system
- `docs/LINKING-RULES.md` — link / back-link conventions
- `docs/WORK_LOG.md` — session-based root log (topic gets registered there)
- `.windsurf/_templates/` — spec + ticket + report + session templates
- `.windsurf/skills/spec-author/SKILL.md` — mindset for authoring specs

## Steps

### 1. Validate

// turbo
- Check that `docs/{TOPIC_SLUG}/` does not already exist
- Read `docs/README.md` to understand existing topics
- If slug already exists, report error and ask user for a different name

### 2. Create Folder Structure

Create the topic folder and required subfolders:
- `docs/{TOPIC_SLUG}/`
- `docs/{TOPIC_SLUG}/specs/`
- `docs/{TOPIC_SLUG}/tickets/`
- `docs/{TOPIC_SLUG}/reports/`

**Conditional subfolders** — do NOT create empty folders; they materialize on
demand:

- `research/` — created only if research will happen in this repo (e.g. via
  `/topic-research`). If research is produced externally (e.g. Traycer),
  link it from `specs/` or `README.md` references instead.
- `stakeholder-reports/` — created the first time `/stakeholder-report` is
  run for this topic. Holds non-technical updates for clients / partners /
  investors. Never auto-generated by the lifecycle.

### 3. Create README.md

**`docs/{TOPIC_SLUG}/README.md`:**
```markdown
# {TOPIC_TITLE}

[← Back to root docs](../README.md)

{TOPIC_DESCRIPTION}

**Branch:** `{suggest a branch name}`
**Status:** Planning

---

## Status

| Phase | Tickets | Status |
|-------|---------|--------|
| _Phases will be added as tickets are created_ | — | — |

---

## Quick Links

- [Changelog](CHANGELOG.md)
- [Root WORK_LOG](../WORK_LOG.md) — session-based project log

### Specs

_Canonical 5-spec set will appear here after `/topic-specs-from-research`:_

- Epic Brief
- Codebase Investigation
- Architecture Spec
- Core Flows
- Implementation Spec
- Execution Plan

### Tickets & Reports

| Ticket | Description | Report | Verification |
|--------|-------------|--------|--------------|
| _Tickets will be added as work is planned_ | — | — | — |
```

### 4. Create CHANGELOG.md

**`docs/{TOPIC_SLUG}/CHANGELOG.md`:**
```markdown
# Changelog — {TOPIC_TITLE}

All notable changes for this topic are documented here.
Each entry references the ticket spec, completion report, and git commit(s).

---

## {YYYY-MM-DD}

### Topic created
- Documentation structure initialized
```

### 5. Register in Hub

Update `docs/README.md` — add the new topic to the Topics table:

```markdown
| [{TOPIC_TITLE}]({TOPIC_SLUG}/) | {TOPIC_DESCRIPTION} | Planning |
```

### 5b. Register in Root WORK_LOG

Add a new topic section at the top of `docs/WORK_LOG.md` using
`.windsurf/_templates/work-log-entry-template.md`:

```markdown
# ({YYYY-MM-DD} — ongoing) {TOPIC_TITLE} ({TOPIC_SLUG}) — 🔄 In Progress

[Detailed plan](./{TOPIC_SLUG}/README.md)

{TOPIC_DESCRIPTION}

## Phases

- [ ] _Phases will be added as tickets are created_

## Sessions

_Session entries will be appended by `/log-session`._

---
```

If `docs/WORK_LOG.md` doesn't exist yet, create it with a top-level heading
`# WORK_LOG` and the topic section underneath.

### 6. Report

```
✅ Topic Created: {TOPIC_TITLE}

📁 Structure:
- docs/{TOPIC_SLUG}/README.md
- docs/{TOPIC_SLUG}/CHANGELOG.md
- docs/{TOPIC_SLUG}/specs/
- docs/{TOPIC_SLUG}/tickets/
- docs/{TOPIC_SLUG}/reports/
- docs/{TOPIC_SLUG}/research/              (conditional — created only if used)
- docs/{TOPIC_SLUG}/stakeholder-reports/   (conditional — created on first /stakeholder-report)

📊 Registered in:
- docs/README.md (Topics table)
- docs/WORK_LOG.md (topic section with phases + sessions subheadings)

🎯 Next Steps:
1. Run research (in Cascade or Traycer)
2. Run /topic-specs-from-research to generate 5-spec set + tickets
3. Start working: /kickoff {first_ticket_id}
```

---

## Notes

- This creates the STRUCTURE only — no tickets, no research
- The topic structure follows the layout defined in this workflow (see "Topic Structure" section above)
- Each topic is self-contained with its own changelog, tickets, and reports
- For work on a topic, use `/kickoff` to work on tickets of the topic
- For other topics, the same ticket-based workflow pattern can be adapted
- Topics can have their own branch or share one — user decides
