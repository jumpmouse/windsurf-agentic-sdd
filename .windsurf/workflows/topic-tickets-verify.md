# TOPIC TICKETS VERIFY — Pre-Implementation Ticket Quality Gate

## 🌐 Project Override Hook

Before executing this workflow, check whether the host project provides an override.

**Path** (in the host project's workspace root, NOT this global repo):
`.windsurf/overrides/workflows/<same-filename-as-this-file>.md`

If present, apply its mode (frontmatter `mode:` OR first heading `# REPLACE` / `# EXTEND` / `# MODIFY`):

- **REPLACE** — run ONLY the override; ignore the steps below.
- **EXTEND** — run the steps below as written, then run the override as a final phase (or at the position it specifies).
- **MODIFY** — run the steps below, with the steps the override declares substituted / inserted at the matching positions.

No override → proceed as written. Ambiguous mode → **STOP and ask the user**.

Verifies that the **generated ticket specs** are sound and faithful to the upstream specs **before** `/topic-implement` runs against them. The ticket specs are the hand-off contract to the implementer — this workflow is the quality gate for that contract.

Use `@verification-fixer` mindset throughout — this skill is explicitly for *"verifying a plan against investigation findings and actual code"*.

## When to Use

- Immediately after `/topic-specs-from-research` finishes generating per-ticket files
- Before any `/topic-implement` run on tickets generated by `/topic-specs-from-research`
- When the user adds / edits a ticket file by hand and wants a sanity check

Do **NOT** use this workflow to verify implementation — that's `/topic-verify`.

## Inputs

- **TOPIC_SLUG** (required): the topic folder under `docs/`
- **TICKET_ID** (optional): if provided, verify only this ticket; otherwise verify ALL tickets in `docs/{TOPIC_SLUG}/tickets/`

## Steps

### 0. Load Skill

// turbo
**MANDATORY — Do this FIRST before any other work.**
Read and internalize the skill file:
- Read `.windsurf/skills/verification-fixer/SKILL.md`

### 1. Read Upstream Specs

// turbo
Read the canonical 5-spec set + plan that the tickets must trace back to:

- `docs/{TOPIC_SLUG}/specs/Execution_Plan_—_*.md` (the source of truth for ticket inventory, ordering, sizing, and dependencies)
- `docs/{TOPIC_SLUG}/specs/Implementation_Spec_*—_File-Level_Change_Map_*.md` (file-level change map)
- `docs/{TOPIC_SLUG}/specs/Architecture_Spec_*.md` (architectural decisions + ownership)
- `docs/{TOPIC_SLUG}/specs/Core_Flows_—_*.md` (user-facing flows)
- `docs/{TOPIC_SLUG}/specs/Epic_Brief_—_*.md` (scope contract)

If any of these files are missing, **STOP** and report — `/topic-specs-from-research` did not finish cleanly.

### 2. Enumerate Ticket Files

If TICKET_ID was provided, the target set is `{docs/{TOPIC_SLUG}/tickets/{TICKET_ID}__*.md}`.
Otherwise, the target set is every `T-*__*.md` file in `docs/{TOPIC_SLUG}/tickets/`.

Cross-check:
- Every entry in Execution Plan §6 (ticket inventory) has a corresponding ticket file ✅
- Every ticket file appears in Execution Plan §6 ✅
- Ticket IDs are unique and contiguous (no gaps unless explicitly justified)

Discrepancies found here are 🔴 **Critical** — they mean the ticket set does not match the plan.

### 3. Per-Ticket Verification

For each ticket in the target set, verify against the upstream specs:

| Check | Rule | Severity if violated |
|-------|------|----------------------|
| **Ticket header** | ID, title, scope contract present and match Execution Plan §6 entry | 🟡 Major |
| **Detailed Steps** | Every step references a `file:` path that exists in Implementation Spec §4 (NEW or MODIFIED) | 🔴 Critical |
| **Acceptance Criteria** | Each AC is testable, traces to a Core Flow or an Architecture decision, matches the AC seed in Execution Plan §7 | 🟡 Major |
| **Guardrails** | Match the scope contract (no out-of-scope file paths, no "fix everything" wording) | 🟡 Major |
| **Dependencies — "Can Start After"** | Match Execution Plan §4 dependency map exactly | 🔴 Critical |
| **Dependencies — "Can Run In Parallel With"** | Bidirectional with the sibling ticket's parallel claim | 🟡 Major |
| **Sizing** | Total touched lines is within budget (~100–500 lines, or explicit exception noted) | 🔵 Minor |
| **File existence** | Every `file:` ref in the ticket resolves on disk (`ls` / `find`) | 🔴 Critical |

Track per-ticket findings — feeds §5.

### 4. Cross-Ticket Verification

Beyond per-ticket checks, verify the **set as a whole**:

- **Coverage**: every NEW / MODIFIED file in Implementation Spec §4 is owned by exactly ONE ticket. No file is split across tickets unless the split is explicitly noted in Execution Plan §3.
- **Ordering**: there is no cycle in the "Can Start After" graph (topological sort succeeds).
- **Sizing rollup**: sum of all per-ticket line budgets matches the total in Execution Plan §5 (within a tolerance — flag if the delta is >20%).
- **No orphans**: no file in Implementation Spec §4 is unowned (no ticket touches it).

Discrepancies in the SET are 🔴 **Critical** — the plan is internally inconsistent.

### 5. Categorize & Decide

Compile findings:

```markdown
## Ticket Verification Findings

🔴 Critical: {count}
🟡 Major: {count}
🔵 Minor: {count}
```

Decision rule:
- **Any 🔴 Critical** → STOP and report. Do NOT auto-fix tickets. Ask user how to resolve (fix tickets in place, or re-run `/topic-specs-from-research` Step 12).
- **Only 🟡 Major / 🔵 Minor** → propose fixes inline (per `@verification-fixer` severity discipline) and wait for user approval before editing ticket files.

### 6. Quality Gate

- [ ] Every ticket in the target set has been read end-to-end
- [ ] Every `file:` reference in every ticket resolves on disk
- [ ] Every ticket maps to an Execution Plan §6 entry (and vice versa)
- [ ] Every "Can Start After" claim matches the plan's dependency map
- [ ] No 🔴 Critical findings remain unresolved
- [ ] All 🟡 Major findings are resolved or deferred with a follow-up note

### 7. Write Tickets-Verification Report

Create `docs/{TOPIC_SLUG}/reports/TICKETS_VERIFICATION_REPORT.md` (or `{TICKET_ID}__TICKET_VERIFICATION_REPORT.md` if a single ticket was verified) using `.windsurf/_templates/verification-report-template.md` adapted for ticket verification:

- Summary + verdict
- Per-ticket check tables (from §3)
- Cross-ticket check results (from §4)
- Findings with severity (from §5)
- Fixes applied (if any)
- Quality gate checklist (from §6)
- Final verdict block (PASS / FAIL / PASS-WITH-FOLLOWUPS)

### 8. Report

```
✅ Tickets Verification Complete — {TOPIC_SLUG}

📊 Results:
- Tickets verified: {N}
- 🔴 Critical: {X}
- 🟡 Major: {X}
- 🔵 Minor: {X}
- Cross-ticket consistency: ✅ / ❌
- Plan ↔ tickets coverage: {X}/{Y}

📄 Report:
  docs/{TOPIC_SLUG}/reports/TICKETS_VERIFICATION_REPORT.md

🎯 Verdict: PASS / FAIL / PASS-WITH-FOLLOWUPS

Next:
- PASS → /kickoff {first_ticket_id} → /topic-implement
- FAIL → fix tickets / re-run /topic-specs-from-research Step 12
- PASS-WITH-FOLLOWUPS → user reviews deferred items, then proceed
```

## Notes

- This workflow **STOPS** after reporting — user reviews before any implementation begins.
- Do NOT auto-edit ticket files for 🔴 Critical findings — surface them and ask the user how to resolve. Critical findings usually mean the upstream specs (Execution Plan / Implementation Spec) need correction, which is upstream of ticket files.
- For 🟡 Major / 🔵 Minor findings, propose specific edits with diffs and wait for user approval (per `@verification-fixer` "no new functionality, fix only what the report says" discipline).
- The Tickets-Verification Report is **distinct** from per-ticket Implementation Verification Reports (`{TICKET_ID}__*__VERIFICATION_REPORT.md`) — those are written by `/topic-verify` AFTER implementation.
- If unable to clear the gate after 3 fix iterations, stop and report the issue.
