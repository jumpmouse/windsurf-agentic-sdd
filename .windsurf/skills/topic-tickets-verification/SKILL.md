# Topic Tickets Verification Skill

## Persona

**MANDATORY first action:** Load the Verifier persona by reading `.windsurf/personas/verifier.md`. Adopt it for the duration of this skill.

## When to use

- Immediately after `/topic-tickets-from-plan` finishes generating per-ticket files.
- Before any `/topic-implement` run on tickets generated by `/topic-tickets-from-plan`.
- When the user adds / edits a ticket file by hand and wants a sanity check.

Do **NOT** use this skill to verify implementation — that's `topic-verification`.

## Inputs

- **TOPIC_SLUG** (required) — topic folder under `docs/`.
- **TICKET_ID** (optional) — if provided, verify only this ticket; otherwise verify ALL tickets in `docs/{TOPIC_SLUG}/tickets/`.

## Output

`docs/{TOPIC_SLUG}/reports/TICKETS_VERIFICATION_REPORT.md` (or `{TICKET_ID}__TICKET_VERIFICATION_REPORT.md` if a single ticket was verified) using `.windsurf/_templates/verification-report-template.md` adapted for ticket verification.

## Procedure

### 1. Read upstream specs

Read the 5-spec set + plan that the tickets must trace back to:

- `docs/{TOPIC_SLUG}/specs/Execution_Plan.md` (source of truth for ticket inventory, ordering, sizing, dependencies)
- `docs/{TOPIC_SLUG}/specs/Implementation_Spec_*—_File-Level_Change_Map_*.md` (file-level change map)
- `docs/{TOPIC_SLUG}/specs/Architecture_Spec_*.md` (architectural decisions + ownership)
- `docs/{TOPIC_SLUG}/specs/Core_Flows_—_*.md` (user-facing flows)
- `docs/{TOPIC_SLUG}/specs/Epic_Brief_—_*.md` (scope contract)

If any of these is missing, **STOP** — the chain (`/topic-specs-from-research` → `/topic-spec-plan` → `/topic-tickets-from-plan`) did not finish cleanly.

### 2. Enumerate ticket files

If `TICKET_ID` provided, target = `{docs/{TOPIC_SLUG}/tickets/{TICKET_ID}__*.md}`. Otherwise target = every `T-*__*.md` in `docs/{TOPIC_SLUG}/tickets/`.

Cross-check:
- Every entry in Execution Plan §6 (ticket inventory) has a corresponding ticket file.
- Every ticket file appears in Execution Plan §6.
- Ticket IDs are unique and contiguous (no gaps unless explicitly justified).

Discrepancies here are 🔴 **Critical** — ticket set does not match the plan.

### 3. Per-ticket verification

For each ticket in target set, verify against upstream specs:

| Check | Rule | Severity |
|-------|------|----------|
| Ticket header | ID, title, scope contract present and match Execution Plan §6 entry | 🟡 Major |
| Detailed Steps | Every step references a `file:` path that exists in Implementation Spec §4 (NEW or MODIFIED) | 🔴 Critical |
| Acceptance Criteria | Each AC is testable, traces to a Core Flow or Architecture decision, matches the AC seed in Execution Plan §7 | 🟡 Major |
| Guardrails | Match scope contract (no out-of-scope file paths, no "fix everything" wording) | 🟡 Major |
| Dependencies — "Can Start After" | Match Execution Plan §4 dependency map exactly | 🔴 Critical |
| Dependencies — "Can Run In Parallel With" | Bidirectional with sibling ticket's parallel claim | 🟡 Major |
| Sizing | Total touched lines within budget (~100-500, or explicit exception noted) | 🔵 Minor |
| File existence | Every `file:` ref in the ticket resolves on disk (`ls` / `find`) | 🔴 Critical |

Track per-ticket findings — feeds §5.

### 4. Cross-ticket verification

Beyond per-ticket checks, verify the **set as a whole**:

- **Coverage**: every NEW / MODIFIED file in Implementation Spec §4 owned by exactly ONE ticket. No file is split across tickets unless explicitly noted in Execution Plan §3.
- **Ordering**: no cycle in "Can Start After" graph (topological sort succeeds).
- **Sizing rollup**: sum of per-ticket line budgets matches total in Execution Plan §5 (within tolerance — flag if delta > 20%).
- **No orphans**: no file in Implementation Spec §4 unowned (no ticket touches it).

Discrepancies here are 🔴 **Critical** — plan is internally inconsistent.

### 5. Categorize & decide

Compile findings:

```markdown
## Ticket Verification Findings

🔴 Critical: {count}
🟡 Major: {count}
🔵 Minor: {count}
```

**Decision rule:**
- **Any 🔴 Critical** → STOP and report. Do NOT auto-fix tickets. Ask user how to resolve (fix in place, re-run `/topic-tickets-from-plan`, or loop further upstream to `/topic-spec-plan` if Execution Plan itself is wrong).
- **Only 🟡 Major / 🔵 Minor** → propose fixes inline (per Verifier persona's severity discipline) and wait for user approval before editing ticket files.

### 6. Quality Gate

- [ ] Every ticket in target set has been read end-to-end.
- [ ] Every `file:` reference in every ticket resolves on disk.
- [ ] Every ticket maps to an Execution Plan §6 entry (and vice versa).
- [ ] Every "Can Start After" claim matches the plan's dependency map.
- [ ] No 🔴 Critical findings remain unresolved.
- [ ] All 🟡 Major findings are resolved or deferred with a follow-up note.

### 7. Write Tickets-Verification Report

Create `docs/{TOPIC_SLUG}/reports/TICKETS_VERIFICATION_REPORT.md` (or single-ticket variant) using `.windsurf/_templates/verification-report-template.md`:

- Summary + verdict
- Per-ticket check tables (from §3)
- Cross-ticket check results (from §4)
- Findings with severity (from §5)
- Fixes applied (if any)
- Quality gate checklist (from §6)
- Final verdict block (PASS / FAIL / PASS-WITH-FOLLOWUPS)

## Final Report

```
✅ Tickets Verification Complete — {TOPIC_SLUG}

📊 Results:
- Tickets verified: {N}
- 🔴 Critical: {X}
- 🟡 Major: {X}
- 🔵 Minor: {X}
- Cross-ticket consistency: ✅ / ❌
- Plan ↔ tickets coverage: {X}/{Y}

📄 Report: docs/{TOPIC_SLUG}/reports/TICKETS_VERIFICATION_REPORT.md

🎯 Verdict: PASS / FAIL / PASS-WITH-FOLLOWUPS

Next:
- PASS → /kickoff {first_ticket_id} → /topic-implement
- FAIL → fix tickets / re-run /topic-tickets-from-plan (or upstream /topic-spec-plan if Execution Plan is at fault)
- PASS-WITH-FOLLOWUPS → user reviews deferred items, then proceed
```

Then **STOP**. Do not auto-edit ticket files for 🔴 Critical findings — surface and ask.
